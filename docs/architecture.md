# ResearchSSH-Next — Architecture

A Windows-first (Android-ready) professional SSH client for research workflows.
The design separates a **Rust security core** from a **C++/Qt 6 Quick (QML) UI**
across a stable C ABI.

## Layering

```
┌──────────────────────────────────────────────────────────────┐
│ QML (app/qml)                                                  │
│  • UI + interaction only                                       │
│  • NEVER touches passwords, private keys, or the FFI           │
│  • talks only to `app` (AppController) via properties/slots    │
└───────────────▲────────────────────────────────────────────────┘
                │ Qt properties / slots / signals
┌───────────────┴────────────────────────────────────────────────┐
│ C++ Qt adapter (app/cpp)                                        │
│  • AppController        orchestration, exposed to QML as `app`  │
│  • ServerListModel      QAbstractListModel (left pane)          │
│  • TerminalViewModel    terminal text buffer (center pane)      │
│  • RemoteFileTreeModel  lazy remote file tree                   │
│  • EditorViewModel      in-app remote text editor state          │
│  • CredentialStore      platform secret storage abstraction     │
│  • RustCoreBridge       the ONLY file that calls the FFI;       │
│                         marshals Rust callbacks to the UI       │
│                         thread via Qt::QueuedConnection         │
└───────────────▲────────────────────────────────────────────────┘
                │ stable C ABI (cbindgen header, opaque handles)
┌───────────────┴────────────────────────────────────────────────┐
│ Rust core (rust-core)                                           │
│  • ffi        C ABI, opaque handles, catch_unwind, batching     │
│  • runtime    Tokio multi-threaded runtime                      │
│  • session    state machine + async driver task                 │
│  • provider   SshProvider trait + factory                       │
│  • providers  mock (shipped) · russh (reserved, feature-gated)  │
│  • fs         FileProvider trait + file-operation results        │
│  • event      SessionEvent + #[repr(C)] RsEvent                 │
│  • secret     zeroizing buffers for credentials                 │
└──────────────────────────────────────────────────────────────┘
```

### Rule enforcement

| Rule | How it is enforced |
|------|--------------------|
| QML never sees secrets/FFI | QML only references `app`; secrets live in C++ `CredentialStore` and go straight to the core's `rscore_session_set_password`. |
| Only the FFI module uses `unsafe` | Crate root has `#![deny(unsafe_code)]`; only `mod ffi` carries `#[allow(unsafe_code)]`. |
| No Rust/C++ objects, exceptions or panics cross the boundary | Only `#[repr(C)]` POD + primitives cross; every entry point is wrapped in `catch_unwind`. |
| Callbacks never touch the UI directly | `RustCoreBridge::onRustEvents` copies the batch and posts it with `Qt::QueuedConnection`; all model/UI mutation happens on the UI thread. |
| Terminal data is batched | The session driver drains all ready provider events into one `Vec`, and the FFI emitter performs exactly one callback per batch. |

## The C ABI

Header is generated by **cbindgen** from `build.rs` into
`rust-core/include/research_ssh_core.h` (committed; regenerated on every
`cargo build`).

### Handles (opaque, Rust-owned)

* `RsCore*` — owns the Tokio runtime. `rscore_create` / `rscore_destroy`.
* `RsSession*` — one session. `rscore_session_create` / `rscore_session_destroy`.

### Functions (minimum surface)

| Function | Purpose |
|----------|---------|
| `rscore_version` | static version string (no free) |
| `rscore_error_message` | static UTF-8 description for an error code (no free) |
| `rscore_string_free` | free a heap string transferred to the caller (reserved) |
| `rscore_create` / `rscore_destroy` | core lifecycle |
| `rscore_session_create` / `rscore_session_destroy` | session lifecycle + event callback registration |
| `rscore_session_connect` / `rscore_session_disconnect` | connect / disconnect |
| `rscore_session_send` | send bytes (batched data) |
| `rscore_session_cancel` | cancel the in-flight operation |
| `rscore_session_set_password` | hand a secret to the provider (copied into a zeroizing buffer) |
| `rscore_session_set_file_callback` | register batched file-operation result delivery |
| `rscore_session_fs_*` | list/read/write/rename/remove/mkdir/copy remote paths |

All fallible calls return an `RsErrorCode` (or null pointer) plus a UTF-8 message
(static via `rscore_error_message`, or per-event via `RsEvent.message`).

### Ownership rules (must be obeyed by the C++ side)

1. **Handles** are heap-allocated and owned by Rust. Store the pointer; call the
   matching `*_destroy` exactly once. Using a handle after destroy is UB.
2. **`RsEvent.data` / `RsEvent.message`** point into Rust-owned buffers that are
   valid **only for the duration of the callback**. Copy them before returning.
   (`RustCoreBridge::onRustEvents` copies into `QByteArray` / `QString`.)
3. **The event callback** may run on any Rust runtime thread. Marshal to the UI
   thread before touching Qt objects.
4. **After `rscore_session_destroy` returns**, the driver task has been joined, so
   no further callbacks can fire. This makes it safe to free the C++ receiver
   afterwards. Destroy sessions before their receiver (`AppController`).
5. **Strings passed in** (`host`, `username`) are borrowed UTF-8; Rust copies them
   during the call. They need not outlive it.

## Event flow (connect → output → disconnect)

```
QML Connect ─► AppController::connectToServer
            ─► RustCoreBridge::createSession + connectSession
                  └─► Rust: session driver, state=Connecting (event)
                         └─► MockProvider::connect → banner bytes (event, batched)
                              state=Connected (event)
   Rust thread ──(RsEvent batch)──► RustCoreBridge::onRustEvents
                  └─ copy + QMetaObject::invokeMethod(Qt::QueuedConnection)
   UI thread   ──► AppController::ingestRustEvents
                  ├─ StateChanged → ServerListModel + status properties
                  ├─ Data         → TerminalViewModel::appendBytes
                  └─ Error        → TerminalViewModel::appendNotice
```

## Threading model

* One Tokio multi-threaded runtime per `RsCore`.
* Each session is driven by exactly one async task (`session::drive`). It owns the
  provider and the state machine, so there is no shared mutable session state.
* Cancellation is delivered out-of-band via a `tokio::sync::Notify` so it can
  interrupt an in-flight `connect`/`send`.
* The FFI callback is invoked serially per session (from the driver task), and
  the C++ side hops to the UI thread before mutating anything.

## Provider abstraction

```rust
#[async_trait]
pub trait SshProvider: Send {
    fn set_secret(&mut self, secret: Secret);
    fn set_host_key_gate(&mut self, gate: HostKeyGate);
    fn take_file_provider(&mut self) -> Option<Box<dyn FileProvider>>;
    async fn connect(&mut self, sink: ProviderSink) -> CoreResult<()>;
    async fn send(&mut self, data: &[u8]) -> CoreResult<()>;
    async fn disconnect(&mut self) -> CoreResult<()>;
}
```

* `MockProvider` (shipped): simulates latency, a banner, command echo with canned
  research responses (`nvidia-smi`, `squeue`, `df`, `python`), and an error path
  (hosts starting with `fail`).
* `RusshProvider` (implemented, shell): `src/providers/russh.rs`, compiled with
  `--features russh`. Real handshake, password auth, host-key confirmation
  (known_hosts + a UI prompt for unknown hosts), PTY + shell, and bulk streaming.
  Uses the pure-Rust `ring` crypto backend. It also attempts to open an SFTP
  subsystem on a second channel; when available, file ops are backed by
  `russh-sftp`.

## Real SSH / SFTP (`russh`)

Real SSH is compiled with `-DRESEARCHSSH_RUST_FEATURES=russh` (CMake) or
`cargo build --features russh`. The backend supports password auth, host-key
confirmation, PTY shell IO and best-effort SFTP. The in-process e2e test verifies
a real SSH handshake/auth/shell over loopback; SFTP availability depends on the
server exposing the subsystem.

Everything above the provider (sessions, FFI, UI) is provider-agnostic.

## Future work (intentionally out of scope for the framework stage)

* VT/ANSI parsing in the Rust core (a `vt` module feeding a real terminal grid).
* File workflow hardening: chunked large-file read/write, progress UI, and
  operation cancel.
* Public-key authentication and richer SSH configuration.
* Port forwarding and multiplexed channels.
* Real `CredentialStore` backends: Windows Credential Manager, Android Keystore.
* Android packaging (`aarch64-linux-android`) — see the README toolchain notes.
